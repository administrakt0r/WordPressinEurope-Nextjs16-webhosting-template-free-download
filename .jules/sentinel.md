## 2026-05-29 - Missing Lightweight WAF for Query Parameters
**Vulnerability:** The application had robust path-blocking, IP rate-limiting, and malicious User-Agent blocking but lacked a mechanism to block common attack vectors passed through query parameters, increasing the risk of unhandled XSS or SQL injection probes reaching the server logic.
**Learning:** Defense-in-depth requires analyzing multiple facets of an incoming request. While the application didn't have explicitly vulnerable database connections directly exposed to user input, intercepting malicious payloads (like `union select`, `<script>`, `../`) at the edge prevents potentially harmful requests from consuming backend resources or triggering unexpected behaviors in less secure downstream services. Furthermore, iterating directly over `URLSearchParams` ensures proper decoding of URL-encoded strings, neutralizing basic obfuscation techniques like using `+` instead of spaces.
**Prevention:** Implement a lightweight Web Application Firewall (WAF) in middleware that parses `request.nextUrl.searchParams` and blocks requests whose keys or values match common attack signatures using a pre-compiled regular expression.
## 2024-03-06 - Prevent URI DoS Attacks
**Vulnerability:** The application was vulnerable to Denial of Service (DoS) attacks via excessively long URIs and query parameters because the Next.js routing/parsing layer could be overwhelmed before custom WAF rules were evaluated.
**Learning:** Checking the `User-Agent` length was not enough. Attackers could send extremely long `pathname` or `search` values, consuming server resources during URL resolution or regex evaluation further down the middleware chain.
**Prevention:** Added explicit length limits (2048 characters) for `request.nextUrl.pathname` and `request.nextUrl.search` early in `middleware.ts`, immediately returning a 414 URI Too Long status. Additionally, applied the same cache-busting headers to 414 responses as 403/429s.
## 2025-05-30 - WAF Double URL Encoding Bypass Prevention
**Vulnerability:** The Lightweight WAF implemented in `isMaliciousQuery` decoded URL parameters only once before checking against malicious signatures. Attackers could theoretically bypass the WAF filter by double-encoding or triple-encoding their malicious payloads (e.g. replacing `<script` with `%253Cscript`).
**Learning:** Security filters evaluating URL inputs must consider encoding bypass vectors. Decoding only once is insufficient when attackers can artificially construct recursively encoded payloads to hide signatures from edge security layers.
**Prevention:** Implement recursive `decodeURIComponent` inside a loop that breaks when the decoded output matches the input (indicating complete decoding) before checking values against regex rules.

## 2025-05-30 - Production Information Leakage via Error Boundaries
**Vulnerability:** Both `app/error.tsx` and `app/global-error.tsx` called `console.error(error)` directly on the client side without checking the environment. This exposed internal error details and potential stack traces in production if unhandled exceptions leaked to the client browser console.
**Learning:** Next.js error boundaries run on the client side, and passing un-sanitized error objects to standard output streams presents an information leakage risk to users casually inspecting the developer console.
**Prevention:** Always wrap un-sanitized diagnostic logging calls like `console.error(error)` in environment condition blocks (`if (process.env.NODE_ENV !== 'production')`) when placed in client-facing components.
## 2025-05-30 - HTTP Method Allowlisting
**Vulnerability:** The application only explicitly blocked \`TRACE\` and \`TRACK\` HTTP methods to prevent XST attacks. However, it did not restrict other potentially harmful or unexpected methods (like \`PUT\`, \`DELETE\`, \`PATCH\`, \`CONNECT\`), leaving the application vulnerable to unexpected behaviors if edge servers or downstream components incorrectly process them.
**Learning:** A deny-list approach (blocking specific bad inputs/methods) is generally less secure than an allow-list approach (only permitting known good inputs/methods). It is safer to explicitly define the allowed methods (\`GET\`, \`HEAD\`, \`POST\`, \`OPTIONS\`) and block everything else.
**Prevention:** Implement explicit allow-listing of HTTP methods in the Edge \`middleware.ts\`, rejecting any unexpected methods with a \`405 Method Not Allowed\` response.
## 2023-11-20 - Outdated Core Framework Dependencies
**Vulnerability:** The application was running Next.js 16.1.6, which was flagged by `pnpm audit` for multiple high and moderate severity vulnerabilities, including HTTP request smuggling in rewrites, unbounded next/image disk cache growth, unbounded postponed resume buffering (DoS), and null origin bypassing Server Actions/HMR CSRF checks.
**Learning:** Security is not just about writing secure code, but also maintaining the security posture of the underlying frameworks and dependencies. Even if the application logic itself is secure, vulnerabilities in the framework can expose the application to significant risks. Regularly auditing and updating dependencies is crucial.
**Prevention:** Implement a process to regularly run `pnpm audit` or use automated dependency scanning tools (like Dependabot or Renovate) to detect and patch vulnerabilities in core frameworks promptly. Keep `next` and `eslint-config-next` updated to the latest stable versions.
