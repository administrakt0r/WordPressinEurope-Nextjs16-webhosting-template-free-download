## 2026-05-29 - Missing Lightweight WAF for Query Parameters
**Vulnerability:** The application had robust path-blocking, IP rate-limiting, and malicious User-Agent blocking but lacked a mechanism to block common attack vectors passed through query parameters, increasing the risk of unhandled XSS or SQL injection probes reaching the server logic.
**Learning:** Defense-in-depth requires analyzing multiple facets of an incoming request. While the application didn't have explicitly vulnerable database connections directly exposed to user input, intercepting malicious payloads (like `union select`, `<script>`, `../`) at the edge prevents potentially harmful requests from consuming backend resources or triggering unexpected behaviors in less secure downstream services. Furthermore, iterating directly over `URLSearchParams` ensures proper decoding of URL-encoded strings, neutralizing basic obfuscation techniques like using `+` instead of spaces.
**Prevention:** Implement a lightweight Web Application Firewall (WAF) in middleware that parses `request.nextUrl.searchParams` and blocks requests whose keys or values match common attack signatures using a pre-compiled regular expression.
## 2024-03-06 - Prevent URI DoS Attacks
**Vulnerability:** The application was vulnerable to Denial of Service (DoS) attacks via excessively long URIs and query parameters because the Next.js routing/parsing layer could be overwhelmed before custom WAF rules were evaluated.
**Learning:** Checking the `User-Agent` length was not enough. Attackers could send extremely long `pathname` or `search` values, consuming server resources during URL resolution or regex evaluation further down the middleware chain.
**Prevention:** Added explicit length limits (2048 characters) for `request.nextUrl.pathname` and `request.nextUrl.search` early in `middleware.ts`, immediately returning a 414 URI Too Long status. Additionally, applied the same cache-busting headers to 414 responses as 403/429s.
